# Knex → Prisma Migration Strategy

## 1. Migration Approach: Staged Adoption

**Recommendation: Prisma for querying first, Knex migrations kept temporarily.**

Prisma will be introduced as the application ORM for all CRUD and business logic
queries. Knex migrations will remain as the schema migration tool during the
transition period because:

- The existing migration history is battle-tested and already deployed
- Prisma's `prisma db pull` (introspection) can generate the schema from the
  live database without replaying migrations
- RLS policies, triggers, functions, and Supabase-specific DDL are best kept in
  raw SQL migrations (Prisma cannot manage these)
- This avoids a risky big-bang cutover

### Phase 1 — Prisma Querying (this PR)
- Add `prisma` + `@prisma/client` dependencies
- Create `prisma/schema.prisma` via introspection + manual refinement
- Create `src/lib/prisma.ts` singleton client
- Validate schema matches live DB with `prisma db pull`
- No query refactoring yet — just infrastructure

### Phase 2 — Service Layer Migration (next)
- Refactor `src/services/*.js` to use Prisma Client instead of Knex
- Replace Knex transaction blocks with `prisma.$transaction()`
- Keep `$queryRaw` / `$executeRaw` for advisory locks and complex SQL
- Migrate API routes from Supabase client queries to Prisma where beneficial

### Phase 3 — Knex Removal (final)
- Remove `knex`, `knexfile.js`, `src/db/index.js`, `src/db/connection.ts`
- Remove `src/db/migrate.ts`
- Move any remaining SQL migrations to `prisma/migrations/` or standalone SQL
- Remove `pg` if no longer needed directly (Prisma bundles its own driver)

## 2. What Stays in SQL

| Concern | Reason |
|---------|--------|
| RLS policies | Prisma has no RLS support; `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` and `CREATE POLICY` must stay in SQL |
| `handle_new_user()` trigger | Supabase Auth integration — fires on `auth.users` INSERT |
| `set_updated_at()` trigger | Applied to 8+ tables; Prisma's `@updatedAt` only works client-side |
| `enforce_max_two_emergency_contacts()` trigger | DB-level constraint enforcement |
| `night_availability` view | Computed view with aggregation |
| CHECK constraints on status enums | Prisma doesn't generate CHECK constraints |
| Partial unique indexes | Prisma doesn't support `WHERE` clauses on `@@unique` |
| Advisory locks (`pg_advisory_xact_lock`) | Raw SQL via `$queryRaw` |
| `FOR UPDATE` / `FOR UPDATE SKIP LOCKED` | Raw SQL via `$queryRaw` |
| Foreign key to `auth.users` | Cross-schema FK; Prisma can't model `auth` schema |

## 3. What Moves to Prisma

| Concern | Benefit |
|---------|---------|
| All application SELECT/INSERT/UPDATE/DELETE | Type-safe queries, auto-completion |
| Transaction blocks in services | `prisma.$transaction()` with interactive transactions |
| TypeScript types (`src/db/schema.ts`) | Auto-generated by `prisma generate` — single source of truth |
| Connection pooling | Prisma manages its own pool; replaces Knex pool config |
| Join queries (children→parent, reservations→blocks) | Prisma relations with `include` / `select` |

## 4. Risks

| Risk | Mitigation |
|------|------------|
| Schema drift between Prisma model and actual DB | Run `prisma db pull` in CI to detect drift; never use `prisma db push` in production |
| Loss of advisory lock patterns | Keep as `$queryRaw`; wrap in helper functions |
| `FOR UPDATE` not supported in Prisma Client | Use `$queryRaw` for pessimistic locking |
| Prisma's `DateTime` vs Postgres `timestamptz` | Prisma handles this correctly; verify in tests |
| Prisma enum vs Postgres enum mismatch | Use Prisma `enum` for app-level enums; keep Postgres `CREATE TYPE` in SQL |
| Connection pool exhaustion during dual-stack | Set Knex pool max to 2 and Prisma pool to 8 during transition |
| Supabase RLS bypass | Prisma connects with `DATABASE_URL` (service role) — RLS only applies to Supabase JS client connections. Server-side Prisma queries bypass RLS by design (same as current Knex behavior). |

## 5. Rollback Plan

1. Prisma is additive — removing it requires only:
   - Delete `prisma/` directory
   - Remove `prisma` and `@prisma/client` from `package.json`
   - Remove `src/lib/prisma.ts`
   - Revert any service files that were migrated
2. Knex remains fully functional throughout Phase 1 and 2
3. No database schema changes are made by this migration
4. Git revert of the migration branch restores the prior state completely

## 7. Refactor Plan for Server-Side DB Access

### Priority 1: Simple CRUD services
| File | Complexity | Notes |
|------|-----------|-------|
| `src/services/config.js` | Low | Simple key-value SELECT with cache |
| `src/app/api/children/route.ts` | Low | Basic CRUD, already simple |
| `src/app/api/bookings/route.ts` | Low | Read-only queries |

### Priority 2: Moderate transactional services
| File | Complexity | Notes |
|------|-----------|-------|
| `src/services/credit.js` | Medium | Transactions with returning clauses |
| `src/billing/subscription-service.ts` | Medium | Upserts, transactions |
| `src/billing/webhooks.ts` | Medium | Idempotent event processing |

### Priority 3: Complex concurrency-critical services
| File | Complexity | Notes |
|------|-----------|-------|
| `src/services/reservation.js` | High | Advisory locks, FOR UPDATE, complex transactions |
| `src/services/enrollment.js` | High | Batch operations, cross-table transactions |
| `src/services/waitlist.js` | High | Advisory locks, FOR UPDATE SKIP LOCKED |
| `src/services/capacity.js` | Medium-High | Raw SQL for capacity calculations |

**Strategy for Priority 3:** Use Prisma interactive transactions (`prisma.$transaction(async (tx) => {...})`) combined with `tx.$queryRaw` for advisory locks and `FOR UPDATE` clauses. This preserves the concurrency guarantees while gaining Prisma's type safety for the surrounding queries.

## 8. End-to-End Test Checklist

### Auth Flows
- [ ] **Signup**: New user creates account → `auth.users` row created → `handle_new_user()` trigger fires → `parents` row created → user redirected to onboarding
- [ ] **Login**: Existing user logs in → session cookie set → middleware validates → dashboard loads
- [ ] **Session persistence**: Refresh page → session maintained → no re-login required
- [ ] **Logout**: User logs out → session cleared → redirect to login

### Dashboard
- [ ] **Dashboard load**: Authenticated user → dashboard renders → parent data displayed
- [ ] **Admin dashboard**: Admin user → admin panel loads → all users visible

### Child Management
- [ ] **Create child**: Fill form → child created → appears in list
- [ ] **Update child**: Edit child details → changes saved
- [ ] **Delete child**: Remove child → cascades properly
- [ ] **Allergies**: Add/edit/remove allergies → action plans linked
- [ ] **Emergency contacts**: Add up to 2 → trigger enforces limit
- [ ] **Authorized pickups**: Add/edit/remove pickup persons

### Reservation / Schedule
- [ ] **View available nights**: Calendar shows capacity → spots_available computed
- [ ] **Reserve a night**: Select night → reservation created → capacity decremented
- [ ] **Cancel reservation**: Cancel → capacity incremented → credit issued if applicable
- [ ] **Double-booking prevention**: Same child, same night → partial unique index blocks
- [ ] **Capacity full**: Night at capacity → waitlist offered

### Waitlist
- [ ] **Join waitlist**: Full night → parent added to waitlist FIFO
- [ ] **Waitlist offer**: Spot opens → first in queue offered → TTL timer starts
- [ ] **Offer expiry**: TTL expires → next in queue offered

### Billing / Payments
- [ ] **Subscription creation**: Plan selected → Stripe subscription created → `subscriptions` row
- [ ] **Webhook processing**: Stripe event → `billing_events` idempotent insert → subscription updated
- [ ] **Plan change**: Mid-cycle change → `pending_plan_changes` queued → applied at cycle boundary
- [ ] **Credits**: Low-enrollment cancellation → credit computed from block snapshot → `credits` row

### Security
- [ ] **RLS enforcement**: User A cannot see User B's children/reservations via Supabase client
- [ ] **Middleware auth**: Unauthenticated request → 401/redirect
- [ ] **Admin-only routes**: Non-admin → 403
- [ ] **API auth**: Missing/invalid token → rejected

## 9. Recommendation

**Adopt Prisma for querying first (Phase 1 + 2), keep Knex migrations temporarily.**

Rationale:
- Zero risk to existing schema — Prisma introspects, doesn't modify
- Immediate developer experience wins (type safety, auto-complete)
- Service-by-service migration allows validation at each step
- Knex migrations remain the deployment mechanism until Prisma migration history is established
- RLS, triggers, and Supabase-specific DDL stay in SQL permanently

The full Knex removal (Phase 3) should happen only after:
1. All services are migrated and tested
2. CI validates Prisma schema matches production
3. A clean Prisma migration baseline is established from the live database
